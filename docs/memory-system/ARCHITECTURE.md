## 设计目标

### 核心目标
1. **自动化** - 减少人工干预，自动完成记忆管理
2. **可靠性** - 数据不丢失，任务不遗漏
3. **可扩展** - 易于添加新功能
4. **可维护** - 代码清晰，文档完善

### 设计原则
- **互补而非替代** - 工具层增强数据层，不修改数据
- **渐进式增强** - 从简单到复杂，逐步完善
- **故障隔离** - 模块独立，单点故障不影响整体

## 双记忆系统

### 三层架构

#### 1. 日记层 (memory/)

**路径**: `~/workspace/memory/YYYY-MM-DD.md`

**用途**:
- 每日完整对话记录
- 技术操作日志
- 系统配置变更
- 重要事件时间线

**特点**:
- 实时追加写入
- 永久保留
- 原始记录，未加工

**格式示例**:
```markdown
# 2026-03-14 Saturday

## 今日概要
- 日期: 2026-03-14
- 主要事件: 完成记忆管理系统 v3.0 重构

## 关键成果
1. Python 重构完成
2. 26 个单元测试通过
3. 推送到 GitHub
```

#### 2. 身份层 (self-improving/)

**路径**: `~/workspace/self-improving/`

**三层结构**:

| 层级 | 文件/目录 | 加载方式 | 内容 |
|------|----------|----------|------|
| 🔥 HOT | `memory.md` | 自动加载 | 用户偏好、活跃模式 |
| 🧠 WARM | `projects/`, `domains/` | 按需加载 | 项目级学习、领域知识 |
| 🧊 COLD | `archive/` | 显式查询 | 归档的旧记忆 |

**升降级规则**:
- 模式使用 3 次/7 天 → 提升到 HOT
- HOT 30 天不用 → 降级到 WARM
- WARM 90 天不用 → 归档到 COLD

#### 3. 任务层 (.learnings/)

**路径**: `~/workspace/.learnings/`

**日志文件**:

| 文件 | 内容 | 触发条件 |
|------|------|----------|
| ERRORS.md | 错误记录 | 命令失败、异常 |
| LEARNINGS.md | 学习记录 | 用户纠正、经验总结 |
| FEATURE_REQUESTS.md | 功能请求 | 用户提出新需求 |

**记录格式**:
```markdown
## [LRN-20260314-001] correction
**Logged**: 2026-03-14T14:12:00+08:00
**Priority**: high
**Status**: pending
**Area**: git

### Summary
强制推送覆盖历史代码

### Details
...
```

## 工具层架构

### v3.0 混合架构

```
┌─────────────────────────────────────────┐
│           工具层 v3.0 架构              │
├─────────────────────────────────────────┤
│                                         │
│  ┌─────────────────────────────────┐   │
│  │         Bash 入口层              │   │
│  │         (run.sh)                 │   │
│  │  • 命令解析                      │   │
│  │  • 环境检查                      │   │
│  │  • 锁管理                        │   │
│  │  • 调用 Python                   │   │
│  └─────────────────────────────────┘   │
│                   ↓                     │
│  ┌─────────────────────────────────┐   │
│  │        Python 核心层             │   │
│  │    (memory_manager.py)          │   │
│  │  • 命令调度                      │   │
│  │  • 配置管理                      │   │
│  │  • 模块协调                      │   │
│  └─────────────────────────────────┘   │
│                   ↓                     │
│  ┌─────────────────────────────────┐   │
│  │         功能模块层               │   │
│  │      (modules/*.py)             │   │
│  │  • reader.py    - 数据读取      │   │
│  │  • analyzer.py  - 分析提取      │   │
│  │  • reporter.py  - 报告生成      │   │
│  │  • archiver.py  - 归档维护      │   │
│  │  • health.py    - 健康检查      │   │
│  └─────────────────────────────────┘   │
│                                         │
└─────────────────────────────────────────┘
```

### 为什么选择混合架构

| 方案 | 优点 | 缺点 | 适用场景 |
|------|------|------|----------|
| **纯 Bash** | 简单、无依赖 | 功能受限、难维护 | 简单脚本 |
| **纯 Python** | 功能强大 | 依赖多、启动慢 | 复杂应用 |
| **Bash + Python** ✅ | 兼顾简单和功能 | 稍复杂 | 我们的场景 |

**我们的选择**:
- **Bash 入口**: 处理命令行、环境检查、锁管理
- **Python 核心**: 处理复杂逻辑、数据分析、报告生成

## 模块设计

### 模块列表

| 模块 | 职责 | 关键类/函数 |
|------|------|------------|
| **Reader** | 读取双记忆数据 | `MemoryReader`, `SessionExtractor` |
| **Analyzer** | 分析提取内容 | `MemoryAnalyzer`, `TodoExtractor` |
| **Reporter** | 生成报告 | `MemoryReporter`, `ReportFormatter` |
| **Archiver** | 归档维护 | `MemoryArchiver`, `BackupManager` |
| **Health** | 健康检查 | `HealthChecker`, `DiskChecker` |

### 模块依赖关系

```
Reader → Analyzer → Reporter
   ↓         ↓          ↓
   └────→ Archiver ←───┘
   ↓
Health (监控所有模块)
```

### 设计原则

1. **单一职责** - 每个模块只做一件事
2. **依赖注入** - 通过配置传入依赖，便于测试
3. **错误处理** - 每个模块独立处理错误，不传播异常
4. **日志记录** - 详细记录操作日志，便于排查问题

## 数据流

### 每日归档流程

```
1. 定时触发 (00:00)
   ↓
2. 读取昨天记忆文件
   ↓
3. 分析内容（提取待办、关键词）
   ↓
4. 生成归档标记
   ↓
5. 追加到长期记忆
   ↓
6. 整理今天待办
   ↓
7. 完成
```

### 晨报生成流程

```
1. 定时触发 (08:00)
   ↓
2. 读取昨天记忆
   ↓
3. 统计关键指标
   - 事件数量
   - 待办完成率
   - 学习记录数
   ↓
4. 提取待办事项
   - 按优先级排序
   - 添加图标标记
   ↓
5. 生成晨报 Markdown
   ↓
6. 保存到 reports/
   ↓
7. 发送到飞书（可选）
```

### 健康检查流程

```
1. 定时触发 (04:00)
   ↓
2. 检查日记层
   - 文件数量
   - 最后更新时间
   - 文件大小
   ↓
3. 检查身份层
   - HOT 记忆大小
   - WARM 项目数量
   ↓
4. 检查任务层
   - 待处理错误数
   - 待处理学习数
   ↓
5. 检查磁盘空间
   ↓
6. 生成健康报告
   ↓
7. 告警（如有问题）
```

## 定时任务

### Cron 配置

```bash
# memory-daily: 每日归档（4次/天）
0 0,6,12,18 * * * bash /path/to/run.sh daily

# memory-index: 每周索引（周一）
0 1 * * 1 bash /path/to/run.sh index

# memory-maintenance: 每日维护
0 2 * * * bash /path/to/run.sh maintenance

# memory-backup: 每日备份
0 3 * * * bash /path/to/run.sh backup

# memory-health: 每日健康检查
0 4 * * * bash /path/to/run.sh health

# memory-report: 每日晨报
0 8 * * * bash /path/to/run.sh report
```

### 任务调度策略

| 任务 | 频率 | 执行条件 | 超时处理 |
|------|------|----------|----------|
| daily | 4次/天 | 无 | 跳过 |
| index | 1次/周 | 周一 | 跳过 |
| maintenance | 1次/天 | 每3天/7天 | 跳过 |
| backup | 1次/天 | 无 | 告警 |
| health | 1次/天 | 无 | 告警 |
| report | 1次/天 | 无 | 告警 |

## 扩展性设计

### 添加新模块

1. 创建 `modules/new_module.py`
2. 实现标准接口
3. 在 `memory_manager.py` 中注册
4. 添加测试

### 添加新命令

1. 在 `run.sh` 中添加命令解析
2. 在 `memory_manager.py` 中实现处理函数
3. 更新帮助文档

### 配置扩展

```yaml
# config.yaml
modules:
  reader:
    enabled: true
    options:
      max_files: 100
  
  new_module:
    enabled: true
    options:
      custom_option: value
```
